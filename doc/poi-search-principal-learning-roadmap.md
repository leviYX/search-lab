# POI 搜索负责人级学习路线图

更新时间：2026-04-11

这份路线图不是“会配搜索引擎”的路线，而是面向希望做到 POI / 地图搜索负责人级别的工程师。目标是让你能独立负责从数据生产、索引构建、查询理解、多路召回、多阶段排序，到评估闭环、实验治理、系统演进的完整链路。

这份路线吸收了 Google Search、Google Maps Platform 和 Google Research 的公开资料，但这里的结构化方法是工程归纳，不是声称拿到了 Google 内部标准。

## 1. 先说结论：原路线的主要不足

现有路线有价值，但对负责人级能力仍然偏轻，主要缺口在 6 个方面：

- 没有把 `POI 数据生产、融合、清洗、去重、入库、更新` 当成一级模块。
- 没有把 `Suggest、Query Rewrite、意图识别、别名/同义词治理` 当成主链路。
- 没有把 `多路召回、召回路由、召回融合` 独立展开。
- 没有系统讲清楚 `粗排、精排、业务重排` 的职责边界。
- 没有把 `路线搜索、nearby、viewport、行政区约束` 单独提升到专项层级。
- 没有把 `点击偏差、样本偏差、训练闭环、线上实验治理` 提到足够重要的位置。

如果你在真实工作里已经碰到 POI 构建、数据融合、召回、排序、意图识别、沿路搜这些问题，那么这几个模块必须前置，不能只在后期轻描淡写带过。

## 2. 顶级 POI 搜索工程师必须具备的 10 个模块

1. 搜索质量观与评估体系
2. POI 数据底座与实体治理
3. 索引设计与检索引擎原理
4. Query 理解、Suggest 与 Rewrite
5. 本地搜索核心场景：POI 搜、地址搜、nearby、along-route
6. 多路召回与召回路由
7. 粗排
8. 精排
9. 业务重排与展示层治理
10. 日志回流、训练闭环、实验治理与系统架构

这 10 个模块里，`2/4/5/6/7/8/9/10` 才是你和普通搜索工程师真正拉开差距的地方。

## 3. 完整链路图

建议你先在脑子里固定这条主链路：

`数据接入 -> 清洗融合 -> 实体治理 -> 索引构建 -> Query 理解 -> 多路召回 -> 粗排 -> 精排 -> 业务重排 -> 展示 -> 日志回流 -> 标注/训练 -> A/B 实验 -> 灰度/回滚`

只要线上出现 badcase，你都必须能把问题定位到这条链路中的一个或多个环节。

## 4. 哪些地方需要模型，哪些地方先规则后模型

## 4.1 模型使用总原则

- 不是所有地方都必须上模型。
- 能用规则稳定解决的问题，不要过早复杂化。
- 但到了 `精排、意图识别、Query 理解、别名挖掘、语义补召回` 这些环节，模型通常会成为主力。
- 顶级搜索系统不是“全模型化”，而是“规则、索引、统计方法、机器学习、多目标约束”共同工作。

## 4.2 各环节模型要求

| 环节 | 是否需要模型 | 建议阶段 | 说明 |
| --- | --- | --- | --- |
| POI 去重 / 融合 | 可选，但中后期建议上 | 中期 | 早期可先规则，后期对同名、别名、连锁门店、跨源融合通常要引入匹配模型 |
| POI 质量分 | 建议上 | 中期 | 可先规则打分，后期用分类/排序模型做质量预测 |
| Query 纠错 | 建议上 | 中期 | 规则词典可做 baseline，复杂拼写纠错一般要统计或模型方法 |
| Query 分类 / 意图识别 | 建议上 | 中期 | 类目词、地址词、附近词、路线词、品牌词，通常需要模型或模型+规则 |
| Suggest 排序 | 建议上 | 中期 | 候选生成可规则化，排序阶段一般要模型 |
| 同义词 / 别名挖掘 | 建议上 | 中后期 | 维护词典不够，长尾别名挖掘通常靠挖掘模型或向量相似 |
| 召回路由 | 可选 | 中后期 | 先规则路由，后期可做 query router 模型 |
| 粗排 | 可选 | 中后期 | 可以先用规则或线性打分，流量上来后常用 LR / GBDT / 小 DNN |
| 精排 | 强烈建议上 | 中后期 | 这是排序模型主战场，常见是 GBDT / LambdaMART / DNN / reranker |
| 业务重排 | 不以模型为主 | 全阶段 | 业务约束、营业状态、去重、多样性、商业规则通常是规则主导，模型辅助 |
| 语义补召回 | 可选 | 中后期 | 向量召回适合长尾词、别名、语义表达，不适合作为地图搜索唯一主链 |
| 反馈闭环与训练 | 必须上 | 后期 | 负责人级别必须掌握训练样本、偏差校正、实验评估闭环 |

## 5. 12 个月负责人级学习路线

## 第 0 月：基线、评估、数据体检

### 目标

建立工作台，先把评估和数据现状搞清楚，而不是先冲着模型去。

### 学习重点

- 搜索指标：Recall@K、MRR、NDCG、零结果率、首条满意率、会话成功率
- query 分桶
- badcase 归因
- POI 样本抽检
- 数据源清单梳理

### 必做产出

- `200-500` 条 query 离线评估集
- 一份 query taxonomy：`brand/address/category/nearby/route/ambiguous/landmark`
- 一份 POI 数据健康度报告：缺字段率、坐标异常率、营业状态缺失率、地址结构化缺失率、别名覆盖率

### 模型要求

- 暂时不上模型
- 先把评估体系和 badcase 归因跑通

## 第 1 月：POI 数据底座与实体建模

### 目标

把“搜什么”定义清楚。没有好的实体建模，后面所有搜索优化都不稳。

### 学习重点

- POI canonical schema
- 主键设计：`poi_id / source_id / canonical_id`
- 名称、别名、地址、类目、品牌、商圈、营业状态、时间戳、坐标
- 层级关系：行政区、道路、门牌、楼栋、园区、商圈
- 变更类型：新开店、闭店、搬迁、改名、合并

### 必做产出

- 一份 POI 实体模型文档
- 一份字段字典与枚举规范
- 一份数据更新策略：全量、增量、回补、回滚

### 模型要求

- 不要求上模型
- 这一阶段以 schema 设计、数据理解、样本抽检为主

## 第 2 月：POI 融合、清洗、去重、质量治理

### 目标

建立“实体可信度”。顶级搜索的前提不是排序强，而是底层实体足够干净。

### 学习重点

- 多源融合
- 规则去重
- 同名判定与错合并风险
- 地址归一化
- 品牌归一化
- 类目归一化
- 质量分体系

### 必做产出

- 一套 merge policy：来源优先级、字段冲突处理、时间优先级
- 一套 duplicate policy：同名同坐标、同名近距离、别名匹配、品牌约束
- 一套质量分规则：名称完整度、地址完整度、坐标可信度、营业状态可信度、类目可信度

### 模型要求

- 早期可先规则
- 中期建议引入模型处理：
  - 重复实体判定模型
  - POI 质量分类模型
  - 地址可信度模型

## 第 3 月：索引设计与基础检索

### 目标

理解 Lucene / ES / OpenSearch 的索引和检索机制，并用 POI 数据做一版可靠 baseline。

### 学习重点

- analyzer、tokenizer、token filter
- 多字段索引设计
- BM25 / BM25F 思想
- geo_point、geo_distance、geo_shape
- refresh、segment、merge
- explain、profile

### 必做产出

- 一个 POI baseline 索引，至少包含 `name/alias/address/category/brand/location/city/district/status`
- 一份不同字段 analyzer 设计说明
- 一份 explain 级别 badcase 分析

### 模型要求

- 暂时不用模型
- 先把词法检索和字段权重做实

## 第 4 月：Query 理解、纠错、Suggest

### 目标

把 query 端的工作做深。很多人把问题都甩给排序，这是错的。

### 学习重点

- 规范化：全半角、大小写、空格、数字、符号、拼音、简称
- 拼写纠错
- Query 分类
- 实体识别
- 槽位抽取：行政区、地点、类目、品牌、路线词、附近词
- 自动补全
- Query Rewrite

### 必做产出

- 一套 query parser
- 一套 suggest pipeline：候选生成、去重、排序、截断、兜底
- 一份 query rewrite 规则清单

### 模型要求

- 纠错：建议用统计方法或模型
- Query 分类 / 意图识别：建议上分类模型
- Suggest 排序：建议上轻量排序模型
- 规则仍然必须保留，尤其在地址词、品牌词、行政区词上

## 第 5 月：别名、同义词、品牌词、类目检索

### 目标

解决本地搜索里最常见、最难维护的语义映射问题。

### 学习重点

- 别名治理
- 品牌词与门店词关系
- 同义词边界
- 类目树设计
- 类目召回与类目排序
- 同名 POI 区分

### 必做产出

- 一份 alias / synonym / brand dictionary 规范
- 一份品牌主实体与门店实体的展示策略
- 一份类目树与类目检索策略说明

### 模型要求

- 长尾别名挖掘建议上模型
- 同义词发现建议上统计或向量方法
- 类目识别建议上分类模型
- 品牌词主实体识别可做模型，但早期也可规则化

## 第 6 月：地址搜索、地理编码、nearby、along-route

### 目标

把地图搜索真正拉开差距的空间能力做透。

### 学习重点

- free-form address 与 structured address
- 地址解析
- 地址归一化
- geocoding / reverse geocoding
- nearby 检索
- viewport / region restriction
- along-route 检索
- detour cost

### 必做产出

- 一个地址搜索实验链路
- 一个 nearby 搜 baseline
- 一个 along-route 搜 baseline
- 一份 `address/nearby/route` 三类 query 的差异化策略

### 模型要求

- 地址 parser 可先规则后模型
- 地址置信度建议上模型
- nearby 排序可先规则
- route detour ranking 中后期可引入模型

## 第 7 月：多路召回与召回路由

### 目标

建立“不是一个召回，而是多路召回协同”的体系。

### 学习重点

- lexical recall
- alias recall
- category recall
- brand recall
- geo recall
- address structured recall
- vector recall
- recall merge
- query routing

### 必做产出

- 至少 5 路召回链路
- 一份召回路由策略表
- 一份召回覆盖率分析

### 模型要求

- 召回路由可以先规则化
- 长尾 query 的补召回建议加向量模型
- alias recall 和 semantic recall 中后期建议模型化

## 第 8 月：粗排

### 目标

把大候选集快速缩到一个高质量候选集，不误杀核心结果。

### 学习重点

- 粗排职责：高召回、低延迟、低误杀
- 轻量特征设计
- 过滤、打分、阈值
- 多召回通道归一化
- 粗排可解释性

### 典型特征

- 文本匹配分
- 名称精确匹配
- 地址匹配度
- 类目匹配
- 距离
- 行政区一致性
- POI 质量分
- 营业状态
- 热度

### 必做产出

- 一个粗排 baseline
- 一份粗排误杀分析
- 一份粗排延迟预算文档

### 模型要求

- 可先不用模型，先做线性打分或规则
- 中后期建议上轻量模型：
  - LR
  - GBDT
  - 小型 DNN

## 第 9 月：精排

### 目标

把真正复杂的相关性判断交给模型，处理特征交互和多目标平衡。

### 学习重点

- LTR：pointwise / pairwise / listwise
- 特征工程
- 标签设计
- 点击偏差
- 位置偏差
- query 分桶建模
- TopK rerank

### 典型特征

- Query-POI 文本匹配特征
- 别名命中特征
- 类目一致性
- 距离与区域约束
- 用户历史行为
- POI 热度与导航率
- 营业状态
- 质量分
- freshness

### 必做产出

- 一份 30 到 80 个排序特征清单
- 一个精排 baseline 模型
- 一份分桶收益报告：品牌词、地址词、nearby、route、类目词分别看

### 模型要求

- 这里强烈建议上模型
- 常见路线：
  - GBDT / LambdaMART baseline
  - DNN 精排
  - TopK cross-encoder rerank

## 第 10 月：业务重排、去重、多样性、展示层策略

### 目标

让排序结果从“相关”变成“可用、可展示、可运营”。

### 学习重点

- 业务重排边界
- 主品牌和分店的展示优先级
- 去重和聚合
- 多样性
- 营业状态和时间敏感性
- 风险内容降权
- 广告/商业规则与自然结果共存

### 必做产出

- 一份 business rerank rulebook
- 一份去重和聚合策略说明
- 一份展示层策略表

### 模型要求

- 业务重排不以模型为主
- 模型可以辅助打分，但最终必须受明确业务约束控制

## 第 11 月：日志回流、训练闭环、偏差校正

### 目标

你要开始具备“让系统自己越来越强”的能力。

### 学习重点

- exposure / click / navigation / conversion 日志设计
- 训练样本构建
- negative sampling
- selection bias
- position bias
- counterfactual learning
- 数据切分
- 模型监控

### 必做产出

- 一份训练样本规范
- 一份偏差分析报告
- 一套离线评估 + 线上实验的联动流程

### 模型要求

- 必须上
- 否则你只能做静态优化，做不到负责人级迭代能力

## 第 12 月：系统架构、专项治理、负责人能力

### 目标

从“会做策略的人”升级到“能负责方向、系统和团队方法论的人”。

### 学习重点

- 索引构建架构
- 流批一体更新
- freshness SLA
- 灰度、回滚、降级
- 缓存、限流、熔断
- 实验治理
- 成本与收益评估
- 专项项目管理

### 必做产出

- 一张完整搜索架构图
- 一份专项方案：目标、评估、风险、灰度、回滚
- 一份负责人级方法论总结

### 模型要求

- 不是单独做模型
- 而是要掌握“模型如何进入生产链路并被治理”

## 6. 每个月都必须持续做的 8 件事

1. 更新 badcase 集
2. 跑一轮离线评估
3. 抽样看日志
4. 分析一类 query 分桶
5. 完成一个最小实验
6. 输出一篇结构化总结
7. 复盘一次失败优化
8. 修正下一阶段路线

## 7. 负责人级别必须掌握的判断标准

你必须逐步能回答下面这些问题：

- 这个 badcase 是数据问题、解析问题、召回问题还是排序问题？
- 这条 query 应该走哪几路召回？
- 这个场景粗排能否不用模型？为什么？
- 这个场景精排为什么必须上模型？
- 业务重排为什么不能只靠模型？
- 地址搜、nearby、along-route 为什么不能共用一套策略？
- 同名和别名为什么要拆开治理？
- 向量召回在哪些 query 上有价值，在哪些 query 上风险更大？
- 线上收益来自哪里，副作用在哪里，回滚条件是什么？

## 9. 最后的建议

如果你的目标只是“会做一点搜索”，原路线已经够用。

如果你的目标是“POI 搜索负责人”甚至“本地搜索高手”，就不能只学：

- IR 基础
- ES 使用
- 一些排序特征
- 一点向量检索

你必须把下面这些能力打通：

- 数据生产与实体治理
- Query 理解与 Suggest
- 多路召回与召回路由
- 粗排、精排、业务重排
- nearby、地址、along-route 三大地图场景
- 训练闭环、偏差治理、实验体系
- 系统架构、灰度、回滚、负责人级项目推进

这才是顶级 POI 搜索的真正门槛。
